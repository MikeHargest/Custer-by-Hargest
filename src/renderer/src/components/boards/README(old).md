# Infinite Whiteboard System (PIXI.js + React)

Этот документ описывает архитектуру и внутреннюю логику работы бесконечного холста (whiteboard).

## 1. Технический стек и графика

Система использует гибридный подход для обеспечения максимальной производительности:

*   **React (`@pixi/react`)**: Управляет «деревом» объектов. Все элементы доски (картинки, текст, фигуры) описываются как декларативные React-компоненты. Это позволяет синхронизировать состояние доски с UI-панелями (слои, свойства объекта).
*   **PIXI.js**: Высокопроизводительный движок на базе WebGL/WebGPU. Он берет на себя всю тяжелую работу по отрисовке тысяч объектов со скоростью 60 FPS.
*   **Refs + RAF (RequestAnimationFrame)**: Для высокочастотных обновлений (перемещение мыши, зум, рисование линий) мы используем `useRef` для хранения промежуточных координат. Это позволяет избежать «тяжелых» ререндеров React на каждый пиксель движения. Синхронизация визуального состояния происходит раз в кадр через RAF.

## 2. Логика инструментов

Инструменты разделены по специализированным хукам:

### Навигация (`useBoardNavigation`)
*   **Панорамирование (Pan)**: Вычисляет `dx/dy` смещение мыши и обновляет `viewport.x/y`.
*   **Зум (Zoom)**: Масштабирование относительно курсора. Координаты вьюпорта пересчитываются так, чтобы точка под курсором оставалась неподвижной при изменении `viewport.scale`.

### Инструменты рисования (`useBoardTools`)
*   **Перо (Pen)**: Накапливает массив точек. Для оптимизации используется упрощение путей, чтобы не перегружать память лишними данными.
*   **Фигуры (Rect)**: Динамический расчет `width/height` на основе стартовой точки клика и текущей позиции курсора.

### Выделение и манипуляция (`useBoardSelection`)
*   **Лассо (Lasso)**: Прямоугольник выделения, который проверяет пересечение границ (AABB) со всеми элементами на доске.
*   **Трансформация**: Использование ограничивающей рамки (Bounding Box) для изменения размера, перемещения и вращения группы выделенных объектов.

## 3. Система хранения данных (.ibo)

Файлы досок сохраняются в формате JSON с расширением `.ibo`. 

### Оптимизация ресурсов (Media Management)
Для предотвращения раздувания файлов (например, хранение 10МБ картинок внутри текста в формате base64):
1.  **Процесс сохранения (`saveIbo`)**: При сохранении главный процесс (Main Process) сканирует данные.
2.  **Извлечение медиа**: Все `data:` ссылки и абсолютные пути к файлам копируются в локальную папку `media`, создаваемую рядом с файлом `.ibo`.
3.  **Относительные пути**: В самом файле `.ibo` длинные данные заменяются на короткие ссылки вида `media/hash_name.png`.

### Процесс загрузки
При открытии файла рендерер считывает относительные пути и преобразует их обратно в полные системные пути `file:///`, чтобы Electron/Chromium мог отобразить локальные ресурсы.

## 4. Особенности реализации

*   **LOD (Level of Detail)**: На больших масштабах система может упрощать отрисовку объектов для поддержания плавности.
*   **Caching**: Текстуры изображений кэшируются в памяти PIXI, чтобы избежать мерцания при повторных рендерах.
*   **History**: Полная поддержка Undo/Redo через стек снимков состояния (в `useBoardHistory`).

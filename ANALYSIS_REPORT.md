# Анализ проекта Antigravity/Cluster

## Обзор

**Название:** Cluster  
**Версия:** 0.3.33  
**Тип:** Electron-приложение с React и TypeScript  
**Цель:** Комплексный инструмент для управления задачами, таймерами, проектами, заметками и досок с интеграцией Google Calendar

## 3. Архитектура Electron

### Main Process (`src/main/index.ts`)

**Ответственности:**
- Управление жизненным циклом окна (BrowserWindow)
- Системное меню и tray-иконка
- IPC-обработчики для файловой системы
- Google Calendar аутентификация и синхронизация
- Управление окнами (включая мини-окна для таймеров)
- Протоколы `board-asset://` и `local-file://` для безопасной загрузки ресурсов

**Ключевые возможности:**
1. **Single Instance Lock** — предотвращение запуска нескольких копий
2. **Workspace Management** — создание/удаление/переименование проектов
3. **File System API** — чтение/запись JSON, Markdown, Board-файлов
4. **Backup System** — автоматическое резервное копирование
5. **Board Cache** — распаковка ZIP-досок в кэш для редактирования
6. **Google OAuth** — авторизация через localhost-сервер
7. **Window Controls** — кастомные кнопки (без рамки окна)

**Особенности реализации:**
- Используется `safeStorage` для шифрования токенов Google
- Retry-логика при записи файлов (EBUSY/EPERM — антивирус/синхронизация)
- Атомарное сохранение досок (через .tmp файлы)
- Lazy-загрузка записей из ZIP через yauzl

### Preload Script (`src/preload/index.ts`)

**Роль:** Мост между main и renderer процессами с контекстной изоляцией.

**Экспортируемое API (window.api):**
- Диалоги (файлы, изображения, workspace)
- Хранилище (get/set)
- Уведомления
- Управление окнами (minimize/maximize/close, always-on-top)
- Мини-окна (события жизненного цикла)
- Файловая система (заметки, доски, проекты)
- Обзор (описания проектов)
- Google Auth и синхронизация
- Кастомный ресайз (для безрамочного окна)

**События (IPC on):**
- `onSyncTimerState` — синхронизация состояний таймеров
- `onActionTimer` — действия с таймерами
- `onTimerFinished` — завершение таймера
- `onMiniWindowClosed` — закрытие мини-окна

### Renderer Process (`src/renderer/src/App.tsx`)

**Архитектура:**
- Монолитный React-компонент (App) с множеством useState/useEffect
- Нет глобального state-менеджера (Redux/Zustand/MobX) — всё локально
- Поднятие состояний (lifting state up) до App
- Использование `useRef` для избежания лишних ререндеров
- Undo/Redo через историю состояний (макс. 30 шагов)

**Маршрутизация:** Внутренняя (через `currentView` state)
- `overview` — обзор проекта
- `pipeline` — канбан-доска
- `timeline` — календарь
- `notes` — заметки
- `clock` — таймеры и будильники

**Особенности:**
- Все представления монтируются одновременно, скрываются через `display: none`
- Это позволяет таймерам продолжать тикать при переключении вкладок
- Auto-save с debounce 500ms
- Очистка legacy-данных при первом запуске в workspace

## 4. Бизнес-логика и компоненты

### 4.1. Управление проектами

**Project (тип):**
- Иерархическая структура (вложенные subprojects)
- Задачи (TaskItem) с вложенными подзадачами
- События (AppEvent) с поддержкой рекуррентности и исключений
- Прогресс (через tasks / manual / pipeline)
- Цвета, иконки, баннеры
- Пути к файлам (notesPath, boardsPath)

**Особенности:**
- Глубокое копирование через `JSON.parse(JSON.stringify())` для истории
- Рекурсивные обходы для обновления/удаления
- Синхронизация с файловой системой (создание реальных папок)

### 4.2. Таймеры и будильники

**TimerData:**
- Часы/минуты/секунды (не только секундомер)
- Назначение на задачи (taskName)
- Звуковые уведомления
- Флаги: isStopwatch, isPinned, isHeaderPinned

**AlarmData:**
- Дата + время (строки, не Date объекты)
- Ежедневные проверки через setInterval (1 сек)
- Автовоспроизведение звука

**Мини-окна:**
- Отдельные BrowserWindow для каждого таймера
- Всегда поверх всех окон
- Синхронизация состояний через IPC

### 4.3. Заметки и Доски

**Markdown-нотки:**
- Хранение как .md (YAML frontmatter) или .json
- Авто-ID, авто-заголовок из первой строки
- Вложенные папки, корзина (trash)
- Бэкапы (автоматические версии в .backups/)

**Доски (Boards):**
- Хранение в ZIP (.board или .ibo)
- Внутри: board.json + assets/
- Кэширование в userData/board-cache/ при открытии
- Элементы: изображения, видео, ссылки, пути, текст, прямоугольники
- Конвертация путей к файлам в `file:///` при открытии
- Обратная конвертация при сохранении

**Особенности архитектуры досок:**
- Двухступенчатое сохранение: write-board-json (быстро) → pack-board (ZIP)
- Поддержка драг-н-дроп, зума, панорамирования
- PIXI.js для рендеринга (WebGL)
- Система привязки (snap targets)

### 4.4. Календарь и Google Sync

**AppEvent:**
- Поддержка рекуррентности (daily/weekly/monthly/yearly/custom)
- Исключения (exceptions) для серий
- Напоминания (reminder.minutesBefore)
- Внешние ID (externalId) для Google
- Статус синхронизации (synced/pending_push/pending_delete/conflict)

**GoogleSyncManager:**
- Двунаправленная синхронизация
- Поиск календаря по совпадению имени с проектом
- Last Write Wins при конфликтах
- Отложенная загрузка syncToken (заглушка)
- Пакетная обработка (до 2500 событий)

**Особенности:**
- Синхронизация запускается вручную (кнопка)
- Проверка authStatus перед синхронизацией
- Обновление всех проектов рекурсивно
- Виртуальные события (экземпляры серии) при перетаскивании

### 4.5. UI Компоненты

**Главные представления:**
- **ProjectOverview** — детальное отображение проекта
- **PipelineView** — канбан-доска стадий
- **CalendarView** — сетка дней/недель/месяцев с событиями
- **NotesView** — список/редактор заметок
- **BoardsView** — PIXI-редактор досок

**Боковая панель (TaskSidebar):**
- Дерево проектов (сворачиваемое)
- Задачи с подзадачами
- События проекта
- Drag-and-drop перемещение
- Выделение цветом
- Редактирование на месте

**Глобальный поиск (GlobalSearchModal):**
- Поиск по проектам, заметкам, задачам
- Быстрый переход (Ctrl+K)

**Настройки (SettingsModal):**
- Тема (цвета)
- Показ FPS, счетчиков задач
- Использование GPU
- Громкость таймеров
- Интервал бэкапа
- Часовой пояс календаря

## 5. Потенциальные проблемы и области для улучшения

### 5.1. Архитектурные проблемы

#### 5.1.1. Отсутствие глобального state-менеджера
**Проблема:** Все состояния хранятся в App.tsx, что приводит к:
- Огромному размеру главного компонента (1734 строки)
- Сложности в разделении ответственности
- Проблемам с производительностью при частых обновлениях
- Сложности тестирования

**Решение:** Внедрить Zustand/Jotai/MobX для разделения состояний:
- `useProjectsStore` — управление проектами
- `useTimersStore` — таймеры и будильники
- `useNotesStore` — заметки и доски
- `useUIStore` — вид, тема, настройки интерфейса

#### 5.1.2. Слишком глубокая вложенность коллбеков
**Проблема:** updateProjectsRecursive, updateTasksRecursive и т.д. создают глубокие цепочки.

**Решение:** Использовать Immer для иммутабельных обновлений:
```typescript
import { produce } from 'immer';

setProjects(produce(draft => {
  const project = draft.find(p => p.id === projectId);
  if (project) project.tasks.push(newTask);
}));
```

#### 5.1.3. Ручное управление подписками IPC
**Проблема:** В App.tsx много `window.api.onX` с ручным удалением слушателей.

**Решение:** Создать хуки-обертки:
```typescript
function useTimerFinished(callback) {
  useEffect(() => {
    const cleanup = window.api.onTimerFinished(callback);
    return cleanup;
  }, [callback]);
}
```

### 5.2. Проблемы с производительностью

#### 5.2.1. Полный ререндер при любом изменении
**Проблема:** Изменение любого таймера/заметки вызывает пересохранение всего workspace.

**Решение:** 
- Дебаунсинг сохранения (уже есть, 500ms)
- Сохранение только измененных частей (патч-апдейты)
- Использование `useMemo` для тяжелых вычислений (allProjects)

#### 5.2.2. setInterval 1 сек для будильников
**Проблема:** Запускается для каждого компонента App, проверяет все проекты.

**Решение:**
- Единый сервис-воркер или фоновый таймер
- Проверка только активных будильников
- Использование Web Workers

#### 5.2.3. PIXI.js и память
**Проблема:** Кэш досок не очищается при закрытии (boards:close-board есть, но не всегда вызывается).

**Решение:**
- Автоматическая очистка кэша по таймауту неактивности
- Ограничение размера кэша (LRU)
- Принудительная сборка мусора при переполнении

### 5.3. Проблемы с надежностью

#### 5.3.1. Потеря данных при ошибках
**Проблема:** Автосохранение каждые 500ms, но ошибки записи игнорируются (возвращается false).

**Решение:**
- Добавить уведомления об ошибках сохранения
- Локальный undo-буфер в памяти
- Резервное копирование перед критическими операциями

#### 5.3.2. Конфликты Google Sync
**Проблема:** Last Write Wins может привести к потере данных.

**Решение:**
- Хранение истории изменений
- Мануальное разрешение конфликтов
- Более умная стратегия (merge для разных полей)

#### 5.3.3. Утечки памяти в Event Listeners
**Проблема:** Много `window.addEventListener` без гарантии очистки.

**Решение:**
- Централизованный менеджер событий
- Тестирование на утечки (Chrome DevTools)
- Автоматическая отписка при размонтировании

### 5.4. UX/UI проблемы

#### 5.4.1. user-select: none на всё приложение
**Проблема:** В main.css и BoardsView добавлен `user-select: none`, что ломает выделение текста в редакторе.

**Решение:**
- Убрать глобальный `user-select: none`
- Добавить только для неинтерактивных элементов
- В редакторе явно указывать `user-select: text`

#### 5.4.2. Отсутствие offline-first
**Проблема:** Нет явной обработки оффлайн-режима.

**Решение:**
- Service Worker для кэширования
- Очередь операций при отсутствии сети
- Локальный-first подход

#### 5.4.3. Медленный старт
**Проблема:** Загрузка всех данных при старте (scanAllNotes читает ZIP-файлы синхронно).

**Решение:**
- Ленивая загрузка
- Прогресс-бар
- Фоновое сканирование

### 5.5. Безопасность

#### 5.5.1. Жестко заданные Google Client ID/Secret
**Проблема:** В коде (googleSyncManager.ts) и в main.ts.

**Решение:**
- Переменные окружения (.env)
- Для Electron использовать `extraResources`

#### 5.5.2. XSS в редакторе
**Проблема:** Tiptap рендерит HTML, dompurify используется не везде.

**Решение:**
- Санитизация всех пользовательских данных
- CSP заголовки

#### 5.5.3. Локальный файловый доступ
**Проблема:** `local-file://` протокол может читать любые файлы.

**Решение:**
- Ограничение только workspace-директорией
- Валидация путей

## 8. Вывод

Проект **Cluster** — это сложное, многофункциональное desktop-приложение с богатым функционалом. Архитектурно оно представляет собой классический Electron + React стек с некоторыми особенностями:

**Плюсы:**
- Многофункциональность (всё в одном приложении)
- Хорошая интеграция с Google Calendar
- Продвинутая работа с файлами (версии, бэкапы)
- Кастомный UI/UX под desktop
- Использование современных технологий (Vite, TypeScript)

**Минусы:**
- Перегруженный главный компонент
- Отсутствие state-менеджмента
- Потенциальные утечки памяти
- Сложность масштабирования
- Некоторые архитектурные технические долги

**Рекомендация:** При добавлении новых фич рефакторить в сторону Zustand/MobX, выносить логику в сервисы, добавлять тесты и улучшать обработку ошибок.
